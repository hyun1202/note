# 프로그래밍 패러다임
- 프로그램을 구성하고 구현하는 사상이나 접근법
- 명령형과 선언형으로 크게 구분됨
## 명령형 프로그래밍
- **어떻게(How) 할 것인지 구체적으로 명령**을 내리는 방식
- 절차지향, 객체지향 등을 포함하며 상태 변화와 제어 흐름을 상세히 기술
	- **절차지향**: 프로시저, 함수를 기반으로 로직을 절차적으로 구성
	- **객체지향**: 데이터와 함수를 하나로 묶은 객체를 중심으로 설계

### 객체지향
- 프로그램을 객체라는 추상화된 단위로 구성하고 각 **객체는 상태와 행동을 갖고 있으며 메시지 교환을 통해 상호작용**한다.
- 캡슐화, 추상화, 상속, 다형성과 같은 특징이 있다.
- **장단점**
	- 장점: 객체라는 단위로 묶이므로 재사용성, 확장성, 유지보수성 우수
	- 단점: 복잡도 증가
#### 객체지향 특징
- 캡슐화: 데이터와 메서드를 하나의 단위로 묶고, 외부에서 직접 접근을 제한하는 것
- 추상화: 공통된 부분을 추출하여 하나의 개념으로 만드는 것
- 상속: 기존 클래스의 속성과 메서드를 새로운 클래스가 물려받는 것
- 다형성: 같은 인터페이스나 메서드가 다양한 형태로 동작하는 것
#### 객체지향 원칙 (SOLID 원칙)
[SOLID 원칙](SOLID%20원칙.md)
- 단일 책임 원칙: 하나의 클래스는 하나의 책임만 가져야 한다.
- 개방 폐쇄 원칙: 변경에는 닫혀있고 확장에는 열려있어야 한다.
- 리스코프 치환 원칙: 자식은 부모를 완전히 대체할 수 있어야 한다.
- 인터페이스 분리 원칙: 클라이언트는 자신이 사용하지 않는 메서드에 의존하면 안된다.
- 의존성 역전 원칙: 고수준 모듈이 저수준 모듈에 의존해서는 안된다.

## 선언형 프로그래밍
- **무엇을(What) 할 것인지를 기술**하고 어떻게(How) 구현, 실행될지는 위임하는 방식
- 필요한 결과를 선언적으로 표현하고 **내부 구현은 추상화**
- 함수형 프로그래밍 등도 선언형 패러다임에 속한다.
- 장단점
	- 장점: 복잡한 로직을 추상화할 수 있어, 비즈니스 로직을 직관적으로 표현하기 쉽다.
	- 단점: 디버깅이 어렵고, 낮은 수준의 최적화나 세밀한 제어가 힘들다.
### 함수형 프로그래밍
- 순수 함수, 불변성, 부수 효과 최소화 등을 핵심으로 하며 선언형 스타일에 가깝다.
- 함수가 일급 시민으로 취급된다.
	- 함수를 인자로 전달하거나 반환할 수 있다. (고차 함수)
- 병렬 처리나 동시성 프로그래밍에서 유리하다.
- **장단점**
	- 장점: 코드 가독성, 병렬 처리, 테스트 용이성
	- 단점: 학습 곡선, **성능**
		- 모든 데이터를 불변하게 다루는 경우 매번 새로운 데이터를 생성해야하므로 메모리 사용량이 늘어날 수 있다. (다만, 언어나 런타임 레벨에서 최적화를 제공할 수 있다.)
#### 특징
1. 순수 함수(Pure Function): 같은 인자를 주면 항상 같은 결과를 반환
2. 부수 효과(Side Effect) 최소화: 상태 변화를 최소화하기 위해 **변수나 객체를 변경하는 것을 지양**
3. 불변성(Immutable State) 지향: 데이터의 **변경이 필요한 경우 새로운 값을 생성하여 사용**
4. 일급 시민(First-class Citizen) 함수: 함수가 일반 값(숫자, 문자열, 객체)과 동일한 지위를 가진다.
	- 인자로 전달, 변수 대입, 함수 반환 (고차 함수)
5. 선언형(Declarative) 접근: 어떻게(How)가 아닌 **무엇(What)**을 계산할지 기술
6. 함수 합성(Composition): 간단한 함수를 조합해 더 복잡한 함수를 만든다.
7. Lazy Evaluation(지연 평가) (선택적 특징): 필요한 시점까지 계산을 미룬다.
